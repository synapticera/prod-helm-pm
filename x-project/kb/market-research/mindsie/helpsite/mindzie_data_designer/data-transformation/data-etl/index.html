Data Transformation

This guide explains the different methods for ingesting data into mindzieStudio to perform process mining and process intelligence analysis.

Overview

mindzieStudio supports multiple data ingestion approaches to accommodate different deployment scenarios, security requirements, and data sources. Whether you have pre-existing event logs or need to transform raw data from source systems, mindzie provides flexible ETL (Extract, Transform, Load) options.

There are two fundamental approaches:

  1. Direct Event Log Upload: Upload finished event logs in supported formats
  2. Data Transformation with mindzieDataDesigner: Transform raw data from source systems into event logs

Data Ingestion Methods

Method 1: Pre-existing Event Log Upload

If you already have a finished event log, you can upload it directly to mindzieStudio.

Supported Formats

  • CSV: Comma-separated values format
  • XES: Extensible Event Stream format (industry standard)
  • BIN: Proprietary mindzie binary format (smaller and faster than other formats)

Upload Options

Option A: User Interface Upload

The simplest method for manual uploads:

  1. Navigate to the dataset upload screen
  2. Select your event log file
  3. Configure the following fields:
    • Case ID: The unique identifier for each case
    • Activity Name: The name of each activity/event
    • Activity Time: The timestamp for each activity
    • Resource (optional): The person or system that performed the activity

Option B: Dataset API Upload

For automated pipelines and integration:

  • Use the dataset API endpoint controller
  • Supports the same formats as UI upload (CSV, XES, BIN)
  • Requires an API endpoint URL and API key
  • Ideal for automated data pipelines

When to Use Pre-existing Event Logs

Use this approach when:

  • You already generate event logs from another application or software
  • You have an existing ETL pipeline that produces event logs
  • You want to manually upload historical event log files
  • You need to integrate event log generation into an automated workflow

Method 2: Data Transformation with mindzieDataDesigner

If you need to transform raw data from source systems (SQL tables, SAP, Oracle, etc.) into event logs, mindzieDataDesigner provides four different connection scenarios.

mindzieDataDesigner connects to approximately 20 different data sources and can connect to any system with query capability.

Scenario 1: CSV File Upload (No Direct Database Access)

Use Case: You don't have direct access to the source system but have exported CSV files.

How It Works:

  1. Export data tables from your source system (e.g., SAP) as CSV files
  2. Upload the CSV files to mindzieDataDesigner
  3. Files are imported into a SQLite database
  4. Query the data as if connected to a live database
  5. Perform ETL transformations consistently

Pros

  • No database credentials required
  • No direct database access needed
  • Works in all deployment scenarios (Desktop, On-Premise, SaaS)
  • Easy to transition to direct database access later (if CSV file names match table names)

Cons

  • CSV export quality issues are common
  • Special characters (especially commas) must be properly escaped
  • Incorrect CSV formatting can cause import failures
  • Time spent cleaning up improperly exported CSV files

Important: When exporting CSV files, ensure proper escaping of special characters. Unescaped commas in comma-separated files will cause data to overflow into adjacent columns, making the file impossible to read correctly.

Deployment Compatibility: Desktop, On-Premise, SaaS

Scenario 2: On-Premise Database with On-Premise Deployment

Use Case: You have an on-premise database and want to deploy mindzieStudio and mindzieDataDesigner on-premise.

How It Works:

  1. Install mindzieDataDesigner and mindzieStudio on-premise
  2. Configure database connection from the on-premise server
  3. Provide read-only database credentials
  4. Data Designer connects directly to the source system

Connection Types

  • SAP: ODBC connection with read-only username and password
  • Oracle: Oracle connection string with credentials
  • SQL Server: SQL Server connection with credentials
  • Other databases: Appropriate connection strings for the specific database

Connection Options

You can connect to:

  • Production database: Direct connection (typically during off-peak hours like nighttime)
  • Sandbox environment: A test or development database
  • Mirrored server: A replica server that is slightly behind production (e.g., 15 minutes)
  • Data warehouse: An on-premise data warehouse where data has been consolidated

Pros

  • Simple and straightforward setup
  • Full access to all databases on the network
  • No special firewall configuration required
  • Direct control over all components

Cons

  • External consultants need VPN or remote access to the on-premise server
  • Port 443 (SSL) must be opened if external users need access to Data Designer UI
  • Requires on-premise infrastructure maintenance
  • More complex collaboration with external teams

Scenario 3: Cloud Database with Cloud Deployment

Use Case: Your data is in the cloud and you want to use mindzie's SaaS environment.

How It Works:

  1. mindzieDataDesigner and mindzieStudio run in mindzie's SaaS servers
  2. Connect to cloud data warehouses (e.g., Snowflake)
  3. All data transformation occurs in the cloud
  4. No on-premise infrastructure required

Pros

  • No on-premise infrastructure to maintain
  • mindzie handles all software updates and maintenance
  • Easy collaboration with external consultants
  • Scalable cloud-based architecture

Cons

  • Requires cloud-based data warehouse
  • Dependent on mindzie's SaaS availability

Scenario 4: Hybrid - On-Premise Database with Cloud SaaS

Use Case: You have an on-premise database but want to use mindzie's SaaS version for easier maintenance.

There are two approaches to this hybrid scenario:

Approach A: Open Database Port to mindzie Servers

How It Works:

  1. Open the database port (e.g., 1433 for SQL Server) in your firewall
  2. Whitelist mindzie's server IP addresses or DNS names
  3. mindzie's SaaS servers connect directly to your on-premise database
  4. All software runs in mindzie's SaaS environment

Pros:

  • Full database access for queries and transformation
  • mindzie maintains all software in SaaS environment
  • No on-premise software maintenance required
  • Once configured, very easy to maintain

Cons:

  • Opening inbound ports often requires IT security approval
  • May not be allowed in highly regulated environments
  • Requires firewall configuration changes
  • IT departments may be hesitant due to security policies

Approach B: Install Data Designer On-Premise, Push to Cloud

How It Works:

  1. Install mindzieDataDesigner on-premise
  2. Data Designer connects to your on-premise database
  3. Transformed data is pushed to mindzieStudio in the cloud
  4. Only mindzieStudio runs in mindzie's SaaS environment

Pros:

  • No inbound ports required in your firewall
  • Outbound traffic is typically allowed by most companies
  • Minimal on-premise software maintenance (only Data Designer)
  • mindzieStudio fully maintained by mindzie
  • More acceptable to IT security teams

Cons:

  • Requires minimal on-premise infrastructure for Data Designer
  • Data Designer software updates needed on-premise
  • Slightly more complex than full SaaS solution

Security and Encryption

All data transfer and storage in mindzie follows strict security protocols:

Data in Transit

  • All connections use HTTPS: Every connection shown in the architecture diagrams uses encrypted HTTPS
  • No unencrypted HTTP allowed: mindzie does not support unencrypted connections
  • SSL/TLS encryption: All API calls, data uploads, and database connections are encrypted

Data at Rest

  • Tenant-specific encryption: Each tenant has its own unique encryption key
  • Encrypted storage: All event logs and source data stored on mindzie servers are encrypted
  • No mindzie access to keys: mindzie personnel do not have access to tenant encryption keys
  • Applies to all deployments: Both on-premise and cloud deployments use tenant-specific encryption

Security Compliance

  • Works with highly regulated environments
  • Supports compliance requirements for financial services, healthcare, and other industries
  • Can accommodate strict IT security policies

Choosing the Right Approach

Use CSV Upload (Scenario 1) When:

  • You don't have database credentials
  • Your IT policy prohibits direct database connections
  • You need a quick proof-of-concept
  • You have small datasets to analyze
  • You work with Desktop version of mindzieStudio

Use On-Premise to On-Premise (Scenario 2) When:

  • You have full control over your infrastructure
  • You want to keep all data on-premise
  • You have dedicated IT resources for maintenance
  • External consultants don't need frequent access
  • Regulatory requirements mandate on-premise deployment

Use Cloud to Cloud (Scenario 3) When:

  • Your data is already in the cloud
  • You want mindzie to handle all maintenance
  • You need easy collaboration with external teams
  • You want to avoid on-premise infrastructure
  • Scalability and availability are priorities

Use Hybrid Approach (Scenario 4) When:

  • Your source data is on-premise
  • You want to use mindzie SaaS for ease of maintenance
  • You need to balance security with convenience
  • Approach A: If your IT policy allows inbound connections with whitelisting
  • Approach B: If your IT policy restricts inbound connections but allows outbound

Best Practices

For CSV File Upload

  1. Validate CSV exports: Test that exported files can be re-imported before using in production
  2. Proper escaping: Ensure special characters (commas, quotes, newlines) are correctly escaped
  3. Consistent naming: Name CSV files the same as table names to ease transition to direct database access
  4. Test with small samples: Validate your ETL transformations with a small dataset first

For Database Connections

  1. Use read-only accounts: Only grant read access to the database user account
  2. Off-peak extraction: Schedule data extraction during off-peak hours to minimize impact
  3. Consider replica servers: Use mirrored or sandbox databases when possible
  4. Monitor connection performance: Track query performance and optimize as needed

For Hybrid Deployments

  1. Document firewall rules: Keep clear documentation of opened ports and whitelisted IPs
  2. Regular security reviews: Periodically review access permissions with your IT security team
  3. Test outbound connectivity: Verify outbound access to mindzie servers before deploying Data Designer
  4. Plan for updates: Schedule Data Designer updates during maintenance windows

General Recommendations

  1. Start simple: Begin with CSV upload for proof-of-concept, then move to direct connections
  2. Plan for scale: Consider future data volume growth when choosing architecture
  3. Involve IT early: Engage your IT and security teams early in the planning process
  4. Test security policies: Verify your chosen approach complies with company security policies
  5. Consider maintenance: Factor in long-term maintenance requirements when choosing deployment

Technical Requirements

Supported Database Connections

mindzieDataDesigner supports approximately 20 different data sources, including:

  • SQL Server
  • Oracle
  • SAP (via ODBC)
  • MySQL
  • PostgreSQL
  • Snowflake
  • Data warehouses
  • Any system with query capability

Network Requirements

  • HTTPS (Port 443): Required for all API and UI connections
  • Database ports: Specific to your database (e.g., 1433 for SQL Server)
  • Outbound connectivity: Required for pushing data to mindzie SaaS
  • Inbound connectivity: Required only for Scenario 4A (direct database access from mindzie SaaS)

Authentication

  • Database credentials: Username and password (read-only recommended)
  • API authentication: API key and endpoint URL for dataset API
  • Tenant isolation: Each tenant has isolated authentication and encryption
